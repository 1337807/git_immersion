Lab 41: Fetching Changes

Goals

  • Learn how to pull changes from a remote repository.

Steps

Execute:

cd ../cloned_hello
git fetch
git hist --all

Output:

$ git fetch
From /Users/jim/Documents/Presentations/git_immersion/auto/hello
   d45fb1b..82b9d03  master     -> origin/master
$ git hist --all
* 82b9d03 2010-12-28 | Changed REAME in original repo (origin/master, origin/HEAD) [Jim Weirich]
* d45fb1b 2010-12-28 | Updated Rakefile (HEAD, origin/greet, master) [Jim Weirich]
* 9567497 2010-12-28 | Hello uses Greeter [Jim Weirich]
* d3247e0 2010-12-28 | Added greeter class [Jim Weirich]
* 399b53b 2010-12-28 | Added README [Jim Weirich]
* 19e2359 2010-12-28 | Added a Rakefile. [Jim Weirich]
* 0530609 2010-12-28 | Moved hello.rb to lib [Jim Weirich]
* 2fc367b 2010-12-28 | Add an author/email comment [Jim Weirich]
* 15fde51 2010-12-28 | Added a comment (v1) [Jim Weirich]
* e76e760 2010-12-28 | Added a default value (v1-beta) [Jim Weirich]
* 2370a1d 2010-12-28 | Using ARGV [Jim Weirich]
* a3f8b17 2010-12-28 | First Commit [Jim Weirich]

At this point the repository has all the commits from the original repository, but they are not integrated into the the cloned repository’s local branches.

Find the “Changed README in original repo” commit in the history above. Notice that the commit includes “origin/master” and “origin/HEAD”.

Now look at the “Updated Rakefile” commit. You will see that it the local master branch points to this commit, not to the new commit that we just fetched.

The upshot of this is that the “git fetch” command will fetch new commits from the remote repository, but it will not merge these commits into the local branches.

Check the README

We can demonstrate that the cloned README is unchanged.

Execute:

cat README

Output:

$ cat README
This is the Hello World example from the git tutorial.

See, no changes.
